# AI 에이전트 워크플로우 행동 규칙

## 1. 스킬 실행 전 필수 확인 사항

### 1.1 스킬 문서 전체 확인
스킬을 실행하기 전에 **반드시** 해당 스킬 문서(SKILL.md)의 모든 섹션을 확인한다:
- 사용법 및 옵션
- 멀티 LLM 분석 모드 (있는 경우)
- Gate 검증 항목
- 내용 검증 (Quality Validation)

### 1.2 작업 유형 판단
작업을 시작하기 전에 다음을 판단한다:

| 작업 유형 | 판단 기준 | 멀티 LLM |
|----------|----------|----------|
| **새 도메인/기능 (MVP)** | 기존에 없는 새로운 기능 개발 | **필수** |
| 기존 기능 수정/보완 | 이미 존재하는 기능 개선 | 선택적 |
| 버그 수정 | 오류 수정 | 불필요 |
| 간단한 변경 | 설정, 텍스트 변경 등 | 불필요 |

## 2. 멀티 LLM 협업 규칙

### 2.1 필수 적용 상황
다음 상황에서는 **반드시** 멀티 LLM 협업을 사용한다:

1. **사용자가 "새로운 도메인", "새 기능", "MVP"를 언급한 경우**
2. **`/design-all` 워크플로우 실행 시**
3. **PRD 워크플로우(0~7단계) 중 research, requirements 단계**
4. **Gate 2 (UI Design) - 항상 필수** (2026-02 추가)

### 2.2 멀티 LLM 실행 방법
```
1. Claude 분석 먼저 수행
2. mcp__zen__chat으로 gpt-5.2 호출 (병렬)
3. mcp__zen__chat으로 gemini-3-pro-preview 호출 (병렬)
4. 3개 결과 종합하여 최종 산출물 생성
```

### 2.3 모델 ID 및 역할

| 모델 ID | 표시명 | 관점 | 분석 항목 |
|---------|--------|------|----------|
| (Claude 자체) | Claude | 종합 | Best Practice, Anti-pattern, 창의적 아이디어 |
| `gpt-5.2` | GPT-5.2 | 기술/기능 | 기술 스택, 아키텍처, Edge Case, 제약조건 |
| `gemini-3-pro-preview` | Gemini-3-Pro | 사용자/비즈니스 | Persona, User Journey, UX 아이디어 |

> **주의**: mcp__zen__chat 호출 시 `model` 파라미터에 정확한 모델 ID를 사용할 것

### 2.4 멀티 LLM 결과 통합 규칙

3개 LLM 결과를 통합할 때 다음 규칙을 따른다:

#### 공통 의견 처리
- 2개 이상 LLM이 동일한 의견 → **그대로 채택**
- 산출물에 "(공통 의견)" 태그 표시

#### 상충 의견 처리
- Claude가 최종 판단 후 결정
- 산출물에 결정 근거 기록:
  ```markdown
  > **의견 상충 해결**: GPT는 A 방식, Gemini는 B 방식 제안.
  > B 방식 채택 - 이유: 프로젝트 기술 스택과 일관성 유지
  ```

#### 고유 아이디어 처리
- 각 LLM의 고유한 아이디어는 **모두 포함**
- 출처 LLM 표시:
  ```markdown
  - 아이디어 A (GPT-5.2)
  - 아이디어 B (Gemini-3-Pro)
  - 아이디어 C (Claude)
  ```

#### 통합 결과 섹션 형식
```markdown
## 멀티 LLM 분석 결과 종합

### 공통 의견
- {의견 1}
- {의견 2}

### 채택된 의견 (상충 해결)
| 항목 | 채택 | 근거 |
|------|------|------|
| {항목} | {GPT/Gemini/Claude} | {근거} |

### 추가 아이디어
- {아이디어} ({출처 LLM})
```

### 2.5 Gate 2 (UI Design) 전용 규칙 (2026-02 추가)

Gate 2에서는 **항상** 멀티 LLM 3-Way Cross Validation을 수행한다.

#### 필수 LLM 역할

| LLM | 역할 | 검증 항목 |
|-----|------|----------|
| **Claude** | Design Lead | 컴포넌트 구조, UI 표준 준수, Responsive 설계 |
| **GPT-5.2** | Technical Validator | 렌더링 성능, 상태 관리, 번들 사이즈, 재사용성 |
| **Gemini-3-Pro** | UX/A11y Validator | WCAG 준수, 터치 타겟, 트렌드 반영, 인지 부하 |

#### 개발자 확인 프로세스

Round 1 완료 후 **항상** 개발자에게 2라운드 진행 여부 확인:

```
AskUserQuestion: "2라운드(역할 반전 비판)을 진행할까요?"
- [A] 전체 진행 - 모든 영역 비판 (+3회 LLM 호출)
- [B] 특정 영역만 - 상충 있는 영역만 선택
- [C] 스킵 - 현재 결과로 진행
```

#### 역할 반전 규칙 (2라운드)

개발자가 [A] 또는 [B] 선택 시, **Claude 주도**로 역할 반전 실행:

| Round 1 역할 | Round 2 역할 (반전) |
|-------------|-------------------|
| Claude (설계) | → GPT가 설계 비판 |
| GPT (기술) | → Gemini가 기술 비판 |
| Gemini (UX) | → Claude가 UX 비판 |

#### 검증 항목 (19개)

| 구분 | 항목 수 | 담당 |
|------|--------|------|
| 기존 (1-11) | 11개 | Claude |
| 추가 (12-15) | 4개 | GPT-5.2 |
| 추가 (16-19) | 4개 | Gemini-3-Pro |

## 3. 불확실할 때의 행동 규칙

### 3.1 사용자에게 먼저 확인
다음 상황에서는 **진행 전에 사용자에게 확인**한다:

- 작업 유형이 "새 도메인"인지 "기존 기능 수정"인지 불분명할 때
- 멀티 LLM 협업 필요 여부가 불분명할 때
- 스킬 옵션 선택이 필요할 때

### 3.2 확인 질문 예시
```
이 작업이 새로운 도메인/기능 개발인지, 기존 기능 수정인지 확인이 필요합니다.
- 새 도메인/기능: 멀티 LLM 협업으로 진행
- 기존 기능 수정: 단일 분석으로 진행

어떤 유형인가요?
```

## 4. 워크플로우 실행 체크리스트

### PRD 워크플로우 실행 전
- [ ] 사용자의 작업 유형 확인 (새 도메인 vs 기존 수정)
- [ ] 해당 스킬 문서의 모든 옵션 확인
- [ ] 멀티 LLM 필요 여부 결정
- [ ] 새 도메인이면 멀티 LLM **기본 적용**

### 각 단계(Gate) 완료 시
- [ ] Gate 검증 항목 모두 확인
- [ ] 내용 검증 (Quality Validation) 수행
- [ ] 멀티 LLM 결과 종합 섹션 포함 (해당 시)

### Gate 완료 보고 시 (필수)
- [ ] **SKILL.md의 검증 항목 테이블을 그대로 인용**하여 보고
- [ ] 각 항목별 통과/실패 여부 명시
- [ ] 임의로 요약하거나 항목 생략 금지

**보고 형식 예시:**
```markdown
## Gate X 검증 결과

| # | 항목 | 유형 | 결과 |
|---|------|------|------|
| 1 | {SKILL.md 항목 그대로} | 자동 | ✅ 통과 |
| 2 | {SKILL.md 항목 그대로} | 수동 | ✅ 통과 |
| 3 | {SKILL.md 항목 그대로} | 자동 | ❌ 실패 - {사유} |
```

## 5. 백그라운드 에이전트 진행 상황 추적

### 5.1 적용 대상
백그라운드 병렬 실행 시에만 적용 (순차 실행은 자동 보고됨)

### 5.2 진행 파일 규칙

**파일 위치**: `prd/{기능명}/.progress.json`

**파일 형식**:
```json
{
  "feature": "{기능명}",
  "total_gates": 9,
  "tasks": [
    { "gate": 0, "skill": "research", "status": "completed" },
    { "gate": 1, "skill": "requirements", "status": "in_progress", "step": "User Story 작성" },
    { "gate": 2, "skill": "ui-design", "status": "pending" }
  ],
  "updated_at": "2026-02-01T10:30:00Z"
}
```

**상태 값**:
| status | 설명 |
|--------|------|
| `pending` | 대기 중 |
| `in_progress` | 진행 중 (step 필드 포함) |
| `completed` | 완료 |
| `failed` | 실패 (error 필드 포함) |

### 5.3 백그라운드 에이전트 필수 행동

1. **각 Gate 시작 시**: status를 `in_progress`로 업데이트
2. **각 Gate 완료 시**: status를 `completed`로 업데이트
3. **오류 발생 시**: status를 `failed`로, error 필드에 사유 기록

### 5.4 진행 상황 확인 보고 형식

사용자가 "진행 상황 확인해줘" 요청 시:

```markdown
## 백그라운드 작업 진행 상황

| # | Gate | Skill | 상태 |
|---|------|-------|------|
| 1/9 | Gate 0 | research | ✅ 완료 |
| 2/9 | Gate 1 | requirements | 🔄 진행 중 - User Story 작성 |
| 3/9 | Gate 2 | ui-design | ⏳ 대기 |
```

## 6. Provisional 항목 처리 규칙

### 6.1 Provisional 항목이란?
표준 문서에 정의되지 않은 **신규 항목**으로, 워크플로우 진행을 위해 임시로 사용하는 항목이다.

| 유형 | 표준 문서 | 예시 |
|------|----------|------|
| UI 컴포넌트 | `docs/ui-standards/UI_STANDARD.md` | EditableGrid, NotificationBadge |
| DB 용어 | `db_dic/dictionary/standards.json` | order_no (수주번호), customer_id |
| 디자인 트렌드 | `docs/ui-standards/UI_STANDARD.md` | [NEW TREND] 태그 항목 |

### 6.2 Provisional 태그 부착 시점

| Gate | Skill | Provisional 발생 |
|------|-------|------------------|
| Gate 2 | `/ui-design` | 신규 컴포넌트: `[PROVISIONAL]`, 신규 트렌드: `[NEW TREND]` |
| Gate 3 | `/data-model` | 신규 DB 용어: `[PROVISIONAL]` |

### 6.3 Provisional 승인 시점

**Gate 7 (`/prd-finalize`)에서 일괄 승인**한다:

1. 모든 Provisional 항목 수집
2. 사용자에게 승인 요청 (아래 형식)
3. 승인 후 표준 문서 자동 업데이트

### 6.4 승인 요청 형식

```markdown
## Provisional 항목 승인 요청

### 컴포넌트
| # | 이름 | 설명 | 승인 |
|---|------|------|------|
| 1 | EditableGrid | 인라인 편집 가능 그리드 | [ ] |

### DB 용어
| # | 컬럼명 | 한글명 | 영문명 | 데이터타입 | 승인 |
|---|--------|--------|--------|-----------|------|
| 1 | order_no | 수주번호 | Order Number | VARCHAR(50) | [ ] |

승인 옵션:
[A] 전체 승인 (표준 문서에 추가)
[P] 프로젝트 한정 (이번만 사용)
[S] 개별 선택
[R] 전체 거부
```

### 6.5 승인 후 자동 업데이트

| 승인 유형 | 처리 |
|----------|------|
| `[A]` 전체 승인 | UI_STANDARD.md, standards.json 자동 업데이트 |
| `[P]` 프로젝트 한정 | PRD 문서에만 기록, 표준 문서 미변경 |
| `[R]` 거부 | 기존 표준 항목으로 대체 필요 |

## 7. Quality Validation 실행 규칙

### 7.1 Quality Validation이란?
각 SKILL-detail.md에 정의된 **산출물 품질 자체 검증** 기준이다.

### 7.2 실행 시점
**각 Gate 산출물 생성 직후**, 다음 Gate로 넘어가기 전에 수행한다.

### 7.3 실행 방법

1. 해당 스킬의 `SKILL-detail.md` 파일에서 "Quality Validation" 섹션 확인
2. 각 검증 항목 기준과 산출물 대조
3. 미달 항목 발견 시:
   - "미달 시 조치" 수행
   - 자동 개선 후 재검증
4. 모든 항목 통과 시 다음 Gate 진행

### 7.4 Quality Validation 보고 형식

```markdown
## Quality Validation 결과

| 검증 항목 | 기준 | 결과 |
|----------|------|------|
| {항목} | {기준} | ✅ 통과 |
| {항목} | {기준} | ⚠️ 개선 후 통과 - {개선 내용} |
| {항목} | {기준} | ❌ 미달 - {사유} |
```

## 8. 금지 행동

1. **스킬 문서를 읽지 않고 스킬 실행 금지**
2. **새 도메인 작업에서 멀티 LLM 생략 금지**
3. **"선택적" 옵션을 "불필요"로 해석 금지**
4. **불확실한 상황에서 임의 판단 금지** (사용자 확인 필수)
5. **Gate 보고 시 SKILL.md 검증 항목을 임의로 요약/생략 금지** (원본 테이블 인용 필수)
6. **Quality Validation 생략 금지** (각 Gate 완료 전 필수 수행)
7. **Provisional 항목 승인 없이 표준 문서 수정 금지**
