## Quality Validation

산출물 생성 후 다음 기준으로 내용 품질을 자체 검증한다.
기준 미달 시 자동 개선 후 재검증한다.

### User Story 검증

| 검증 항목 | 기준 | 미달 시 조치 |
|----------|------|-------------|
| INVEST 원칙 | Independent, Negotiable, Valuable, Estimable, Small, Testable | 원칙 위반 스토리 재작성 |
| AC 개수 | 스토리당 최소 3개 | AC 추가 |
| AC 검증 가능성 | "~해야 한다"가 아닌 "~되어야 한다" 형태 | 검증 가능한 형태로 재작성 |

### 기능 요구사항 검증

| 검증 항목 | 기준 | 미달 시 조치 |
|----------|------|-------------|
| 입출력 명시 | 모든 FR에 입력/출력 정의 | 누락 항목 추가 |
| US 연결 | 모든 FR이 최소 1개 US와 연결 | 관련 US 명시 |
| 중복 검사 | 유사 FR 병합 또는 차별화 | 중복 제거/명확화 |

### Edge Case 검증

| 검증 항목 | 기준 | 미달 시 조치 |
|----------|------|-------------|
| 최소 개수 | 기능당 최소 3개 | Edge Case 추가 |
| 경계값 포함 | null, empty, max 등 경계값 | 경계값 케이스 추가 |
| 처리 방안 구체성 | 추상적 "처리" → 구체적 동작 | 상세화 |

### 일관성 검증

| 검증 항목 | 기준 | 미달 시 조치 |
|----------|------|-------------|
| 용어 일관성 | 동일 개념에 동일 용어 사용 | 용어 통일 및 정의 추가 |
| 우선순위 정합성 | 중요도/긴급도와 우선순위 일치 | 조정 |

## 멀티 LLM 프롬프트

### gpt-5.2 (기술/기능 관점)

```
mcp__zen__chat(
  prompt: "다음 기능 요구사항을 기술적 관점에서 분석해주세요.

[요구사항]: {사용자 요구사항}
[Research 결과]: {00-research.md 요약}

분석 항목:
1. 기능 요구사항 도출 (FR-xxx)
2. Edge Case 식별 (경계값, 예외 상황)
3. 기술적 제약조건
4. 비기능 요구사항 (성능, 보안, 확장성)
5. 구현 복잡도 예상",
  model: "gpt-5.2",
  working_directory_absolute_path: "{프로젝트 경로}",
  thinking_mode: "medium"
)
```

### gemini-3-pro-preview (사용자/비즈니스 관점)


### 결과 종합 기준

| 종합 기준 | 처리 방법 |
|----------|----------|
| 공통 User Story | 필수 반영 |
| 공통 Edge Case | 필수 반영 |
| 고유 FR | 채택 여부 판단 + 근거 |
| 우선순위 차이 | 근거 비교 후 최종 결정 |
