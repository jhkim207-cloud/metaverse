# 서브에이전트 vs 체크포인트: 심층 비교 분석

## 1. 핵심 질문: "별반 다르지 않은가?"

### 공통점 (표면적으로 유사)

| 측면 | 체크포인트 | 서브에이전트 |
|------|-----------|-------------|
| 목적 | 컨텍스트 분리 | 컨텍스트 분리 |
| 정보 전달 | 파일로 전달 | 결과로 반환 |
| 세션 분리 | O | O |

### 근본적 차이점

```
체크포인트: [세션1] → 파일 저장 → [세션2] 파일 로드 → 계속
            (수동 전환, 사용자 개입)

서브에이전트: [메인] → [서브] → 결과 반환 → [메인] 계속
              (자동 위임, 단일 세션 내)
```

**결론**: 표면적으론 유사하나, **작동 방식과 제약사항이 완전히 다름**

## 2. 서브에이전트 방식 상세 분석

### 2.1 서브에이전트의 핵심 특성

```yaml
# 서브에이전트 정의
---
name: prd-research
description: PRD 리서치 전문가. Gate 0 담당.
tools: Read, Grep, Glob, WebSearch, WebFetch
model: sonnet
---

당신은 PRD 리서치 전문가입니다.
Best Practice 조사, Anti-pattern 식별, 창의적 아이디어 생성을 담당합니다.
```

### 2.2 서브에이전트 적용 시 워크플로우

```
[메인 에이전트]
    │
    ├─→ [prd-research 서브에이전트] → 00-research.md 생성 → 요약 반환
    │
    ├─→ [prd-requirements 서브에이전트] → 01-requirements.md 생성 → 요약 반환
    │
    ├─→ [prd-ui-design 서브에이전트] → 02-ui-design.md 생성 → 요약 반환
    │
    ...
    │
    └─→ [prd-implement 서브에이전트] → 코드 생성 → 결과 반환
```

### 2.3 장점

| 장점 | 설명 |
|------|------|
| **컨텍스트 격리** | 각 서브에이전트가 독립 200k 윈도우 보유 |
| **토큰 절약** | 메인에는 요약만 반환 (67% 감소 효과) |
| **단일 세션** | 사용자가 세션 전환 불필요 |
| **자동 위임** | Claude가 알아서 적절한 서브에이전트 호출 |
| **전문화** | 각 서브에이전트가 해당 Gate에 특화 |
| **병렬 실행** | 독립 작업은 병렬 처리 가능 |

### 2.4 제한사항 및 위험

| 제한 | 영향 | 심각도 |
|------|------|--------|
| **서브에이전트 중첩 불가** | 서브에이전트가 다른 서브에이전트 호출 못함 | 중간 |
| **메인으로 결과 반환** | 모든 결과가 메인 컨텍스트로 | **높음** |
| **MCP 도구 제한** | 백그라운드 서브에이전트는 MCP 사용 불가 | 중간 |
| **SKILL 미상속** | 스킬을 명시적으로 주입해야 함 | 중간 |
| **핸드오프 문제** | 메인이 충분한 brief 안 주면 "context amnesia" | **높음** |

## 3. 핵심 문제: "결과가 메인으로 돌아온다"

### 3.1 체크포인트의 장점이 여기서 드러남

```
체크포인트 방식:
[Session 1] Gate 0-1 실행 → CHECKPOINT.md 저장 → /clear
[Session 2] CHECKPOINT.md 로드 (가벼움) → Gate 2-4 실행

→ 이전 세션의 상세 내용은 메인에 없음!
```

```
서브에이전트 방식:
[메인] → [Gate 0 서브] → 결과 반환 (요약이라도 메인에 추가)
      → [Gate 1 서브] → 결과 반환 (또 메인에 추가)
      → [Gate 2 서브] → 결과 반환 (계속 누적...)

→ 서브에이전트 결과가 계속 메인에 쌓임!
```

### 3.2 서브에이전트만으로는 부족한 이유

| Gate | 서브에이전트 결과 크기 | 메인 누적량 |
|------|----------------------|------------|
| 0 | ~500 토큰 (요약) | 500 |
| 1 | ~500 토큰 | 1,000 |
| 2 | ~800 토큰 | 1,800 |
| 3 | ~600 토큰 | 2,400 |
| ... | ... | ... |
| 8 | ~1,000 토큰 | **8,000+** |

**서브에이전트 사용해도 메인 컨텍스트는 계속 증가!**

## 4. 하이브리드 솔루션: 서브에이전트 + 체크포인트

### 4.1 최적 조합

```
┌─────────────────────────────────────────────────────────────┐
│ Session 1: Discovery                                        │
│                                                             │
│ [메인] → [research-subagent] → 00-research.md              │
│       → [requirements-subagent] → 01-requirements.md        │
│       → CHECKPOINT-1.md 저장                                │
│       → 사용자에게 "Session 1 완료" 알림                    │
└─────────────────────────────────────────────────────────────┘
                              ↓ /clear 또는 새 세션
┌─────────────────────────────────────────────────────────────┐
│ Session 2: Design                                           │
│                                                             │
│ [메인] CHECKPOINT-1.md 로드 (가벼움)                        │
│       → [ui-design-subagent] → 02-ui-design.md              │
│       → [data-model-subagent] → 03-data-model.md            │
│       → [api-design-subagent] → 04-api-design.md            │
│       → CHECKPOINT-2.md 저장                                │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 왜 하이브리드인가?

| 문제 | 서브에이전트만 | 체크포인트만 | 하이브리드 |
|------|--------------|-------------|-----------|
| Gate 내 컨텍스트 | ✅ 격리됨 | ❌ 누적 | ✅ 격리됨 |
| 세션 간 컨텍스트 | ❌ 계속 누적 | ✅ 초기화 | ✅ 초기화 |
| 정보 손실 | ⚠️ 요약 손실 | ⚠️ 요약 손실 | ⚠️ 동일 |
| 사용자 경험 | ✅ 자동 | ❌ 수동 전환 | ⚠️ 절충 |
| 세션당 컨텍스트 | 60-70% | 40-50% | **30-40%** |

### 4.3 하이브리드의 이점

```
Session 1에서:
- 서브에이전트가 Gate 0 실행 → 메인에 요약만 (+500)
- 서브에이전트가 Gate 1 실행 → 메인에 요약만 (+500)
- 총 메인 컨텍스트: ~1,000 토큰 (매우 가벼움)
- CHECKPOINT 저장 → /clear

Session 2에서:
- CHECKPOINT 로드 (~200 토큰, 포인터 위주)
- 서브에이전트 3개 실행 → 각각 요약 반환
- 메인 컨텍스트: ~2,000 토큰으로 유지
```

## 5. 정보 손실 문제: 근본적 해결책

### 5.1 "요약하면 손실된다"는 피할 수 없는 사실

**체크포인트든 서브에이전트든:**
- 전체 내용 → 요약 = 정보 손실
- 이건 피할 수 없음

### 5.2 진짜 해결책: "참조 기반 아키텍처"

```
원칙: 정보를 요약하지 말고, 어디에 있는지 알려줘라

[서브에이전트 결과 반환]
❌ "User Story 5개를 정의했습니다: US-001은 관리자가..."

✅ "Gate 1 완료. 결과: prd/notification/01-requirements.md
    - User Stories: 5개 (상세: 섹션 2)
    - Edge Cases: 8개 (상세: 섹션 6)
    - ⚠️ Critical: 알림 최대 1000자 (섹션 4.1)"
```

### 5.3 서브에이전트 결과 반환 규칙

```yaml
---
name: prd-requirements
description: 요구사항 분석 전문가
---

## 결과 반환 형식 (필수)

작업 완료 시 다음 형식으로만 반환:

### 완료 보고
- 생성 파일: {파일 경로}
- Gate 1: {통과/실패} ({n}/{m})

### 핵심 수치 (다음 Gate 필수)
- User Stories: {n}개
- Edge Cases: {n}개
- 성능 목표: {응답시간}ms

### 필수 참조 (다음 Gate에서 읽을 것)
- 01-requirements.md > 섹션 6: Edge Cases
- 01-requirements.md > 섹션 4.1: 성능 요구사항

⚠️ 전체 내용을 반환하지 마시오. 파일 경로만 반환.
```

## 6. 서브에이전트 아키텍처 설계

### 6.1 서브에이전트 목록

| 서브에이전트 | 담당 Gate | 도구 | 모델 |
|-------------|----------|------|------|
| `prd-research` | 0 | Read, WebSearch, WebFetch, mcp__zen__chat | sonnet |
| `prd-requirements` | 1 | Read, Write, Grep | sonnet |
| `prd-ui-design` | 2 | Read, Write, Glob | sonnet |
| `prd-data-model` | 3 | Read, Write, Bash | sonnet |
| `prd-api-design` | 4 | Read, Write | sonnet |
| `prd-impl-plan` | 5 | Read, Write, Glob | haiku |
| `prd-test-design` | 6 | Read, Write | haiku |
| `prd-finalizer` | 7 | Read, Write, Glob | sonnet |
| `prd-implementer` | 8 | Read, Write, Edit, Bash | sonnet |

### 6.2 서브에이전트 간 정보 전달

```
[prd-research]
    │ 반환: "00-research.md 생성완료. 채택 BP 3개."
    ↓
[prd-requirements]
    │ 입력: "00-research.md 참조하여 요구사항 분석"
    │ 반환: "01-requirements.md 생성완료. US 5개, EC 8개."
    ↓
[prd-ui-design]
    │ 입력: "01-requirements.md 참조하여 UI 설계"
    ...
```

**핵심**: 서브에이전트는 이전 산출물 **파일을 직접 읽음** (메인에서 전달 X)

### 6.3 메인 에이전트 역할

```
메인 에이전트 = 오케스트레이터

역할:
1. 서브에이전트 순서대로 호출
2. 각 서브에이전트에 "이전 파일 참조해" 지시
3. 결과 요약 수집 (컨텍스트 최소화)
4. 세션 경계에서 CHECKPOINT 저장
5. Gate 실패 시 재실행 조율
```

```
│       ├──→ Task(prd-research) ──→ 00-research.md               │
│       │         └─ 반환: "완료. BP 3개. 파일 참조."             │
│       │                                                         │

│       │         └─ 입력: "00-research.md 읽고 분석"             │
│       │         └─ 반환: "완료. US 5개, EC 8개. 파일 참조."     │
│       │                                                         │
│       └──→ CHECKPOINT-1.md 저장                                 │
│                                                                 │
│  메인 컨텍스트: ~1,500 토큰 (요약들 + 오케스트레이션)           │

│       │                                                         │
│       ├── CHECKPOINT-1.md 로드 (~200 토큰)                      │
│       │                                                         │
│       ├──→ Task(prd-ui-design) ──→ 02-ui-design.md             │

│       ├──→ Task(prd-api-design) ──→ 04-api-design.md           │
│       │                                                         │
│       └──→ CHECKPOINT-2.md 저장                                 │
│                                                                 │

└─────────────────────────────────────────────────────────────────┘
```

### 7.2 컨텍스트 비교

| 방식 | Session 1 끝 | Session 2 끝 | Session 3 끝 | Session 4 끝 |
|------|-------------|-------------|-------------|-------------|
| **단일 세션 (현재)** | 40% | 75% | 95% | 초과 💥 |
| **체크포인트만** | 40% | 45% | 50% | 60% |
| **서브에이전트만** | 25% | 45% | 65% | 85% |
| **하이브리드** | **15%** | **20%** | **25%** | **35%** ✅ |

### 7.3 구현 복잡도

| 구성요소 | 복잡도 | 우선순위 |
|----------|--------|----------|
| 서브에이전트 9개 정의 | 중간 | 1 |
| 결과 반환 형식 표준화 | 낮음 | 1 |
| CHECKPOINT 구조 | 낮음 | 2 |
| 오케스트레이터 로직 | 중간 | 2 |
| 자동 /clear | 낮음 | 3 |

## 8. 결론

### 질문에 대한 답변

> "서브에이전트를 구축하는 방법은 어때? 체크포인트와 별반 다르지 않나?"

**표면적으론 유사하나, 조합하면 시너지 발생:**

| 측면 | 서브에이전트 | 체크포인트 | 하이브리드 |
|------|-------------|-----------|-----------|
| Gate 내 격리 | ✅ 해결 | ❌ 미해결 | ✅ 해결 |
| 세션 간 누적 | ❌ 미해결 | ✅ 해결 | ✅ 해결 |
| 자동화 | ✅ 자동 | ❌ 수동 | ⚠️ 반자동 |
| 정보 손실 | ⚠️ 요약 손실 | ⚠️ 요약 손실 | ⚠️ 참조로 최소화 |

### 최종 권장

1. **서브에이전트**: 각 Gate를 독립 컨텍스트에서 실행
2. **체크포인트**: 세션 경계에서 상태 저장/초기화
3. **참조 기반 반환**: 요약 대신 파일 경로 + Critical 수치만

**이 조합으로 컨텍스트 사용량 70% 감소 + 정보 손실 최소화**
