## 10. 리뷰어 행동 가이드

### DO (권장)

- 구체적인 피드백 제공
- 코드 개선을 위한 제안 포함
- 칭찬할 부분 언급
- 질문 형식으로 의도 확인

### DON'T (금지)

- 개인 비판
- 모호한 피드백 ("이상해 보임")
- 스타일만 지적 (린터로 해결 가능한 것)
- 리뷰 없이 승인

## 12. 심각도 분류

| 라벨 | 설명 | 예시 |
|------|------|------|
| 🔴 **Blocker** | 머지 불가, 반드시 수정 | 보안 취약점, 버그 |
| 🟠 **Major** | 수정 권장, 설명 필요 | 성능 이슈, 아키텍처 문제 |
| 🟡 **Minor** | 선택적 수정 | 네이밍 개선, 코드 정리 |
| 🟢 **Suggestion** | 참고용 제안 | 대안 제시, 정보 공유 |

## 13. 리뷰 응답 템플릿

### 문제점 지적

```
🔴 [Blocker] SQL Injection 취약점

문제:
- `${}` 구문으로 사용자 입력이 직접 쿼리에 삽입됩니다.

제안:
- `#{}` 구문으로 변경하여 PreparedStatement를 사용하세요.

참고: docs/security/SECURITY_CHECKLIST.md
```

### 개선 제안

```
🟡 [Minor] 가독성 개선 제안

현재:
if (user != null && user.isActive() && user.hasRole("ADMIN")) { ... }

제안:
boolean isActiveAdmin = user != null && user.isActive() && user.hasRole("ADMIN");
if (isActiveAdmin) { ... }

의도를 명확히 드러내는 변수명을 사용하면 가독성이 향상됩니다.
```
