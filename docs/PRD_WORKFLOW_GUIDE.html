📋 1. 워크플로우 개요

이 프로젝트는 2단계 개발 접근법을 사용합니다. 새로운 기능 개발 시 체계적인 PRD(Product Requirements Document) 워크플로우를 통해 MVP(80% 완성)를 먼저 달성하고, 이후 자유로운 대화를 통해 나머지 20% 완성을 진행합니다.

9 총 Gate 수
54 검증 항목
3 LLM 협업
80% MVP 목표

🎯 MVP 단계 (Gate 0~8)

/design-all [기능명] 명령으로 전체 워크플로우를 자동 실행합니다. 9개의 Gate를 순차적으로 통과하며 설계 문서부터 실제 코드까지 생성됩니다.

🔧 완성 단계

MVP 완료 후 버그 수정, UI/UX 개선, 성능 최적화 등은 별도 워크플로우 없이 자유로운 대화를 통해 진행합니다.

🤖 멀티 LLM 협업

새 도메인/기능 개발 시 Claude, GPT-5.2, Gemini-3-Pro 3개 AI가 협력하여 다양한 관점의 분석을 제공합니다.

📝 Provisional 관리

표준에 없는 새로운 컴포넌트나 DB 용어는 [PROVISIONAL] 태그를 붙여 진행하고, Gate 7에서 일괄 승인합니다.

ℹ️ 서브에이전트 아키텍처

각 Gate는 독립적인 서브에이전트로 실행되어 컨텍스트 폭발을 방지합니다. 메인 에이전트는 ~3.5% 컨텍스트만 사용하며, 각 Gate는 독립된 200k 컨텍스트를 활용합니다. 이를 통해 토큰 사용량 67% 감소와 함께 후반 Gate에서의 품질 저하를 방지합니다.

Gate 4: API Design

요구사항을 API 명세로 변환 - RESTful 엔드포인트, DTO, 에러 코드, 비즈니스 로직

  • 03-data-model.md
📤 산출물
  • prd/{기능명}/04-api-design.md
✅ 검증 항목 (5개)
  • URL 규칙 (소문자, 복수형, 하이픈)
  • HTTP 메서드 일치 (CRUD 패턴)
  • 에러 코드 패턴 (AUTH_/VAL_/BIZ_/SYS_)
  • 검증 규칙 정의 (@Valid)
  • 비즈니스 로직 흐름도
🚨 에러 코드 체계
  • AUTH_: 인증/인가 (401/403)
  • VAL_: 입력값 검증 (400)
  • BIZ_: 비즈니스 로직 (404/409)
  • SYS_: 시스템 (500)

Quality Validation

각 Gate 산출물 생성 후, 다음 Gate로 넘어가기 전에 품질 자체 검증을 수행합니다.

검증 프로세스

1. SKILL-detail.md의 "Quality Validation" 섹션 확인
2. 각 검증 항목과 산출물 대조
3. 미달 항목 자동 개선 후 재검증
4. 모든 항목 통과 시 다음 Gate 진행

검증 유형

자동 형식, 문법, 개수 등 자동 검증 가능 항목
수동 논리적 타당성, 완성도 등 수동 확인 필요 항목

80% 완성도 판정

// 완성도 계산 완성도 = (통과 항목 수 / 총 항목 수) × 100 // 설계 단계 기준 (Gate 0~7) 총 항목 수: 46개 80% 기준: 37개 이상 통과 // 판정 결과 ≥ 80% (37+) → 통과 → /implement 실행 가능 < 80% → 미달 → 실패 Gate 재실행 필요

전체 워크플로우 실행

# 전체 워크플로우 실행 (Gate 0~8) /design-all [기능명] # 특정 단계부터 재개 /design-all [기능명] --from [단계번호] # 예시 /design-all user-management /design-all notification-system --from 3

개별 스킬 실행

Gate 명령어 옵션
0 /research [기능명] --no-multi-llm
1 /requirements [기능명] --no-multi-llm
2 /ui-design [기능명] -
3 /data-model [기능명] --fix, --blocking
4 /api-design [기능명] --fix
5 /impl-plan [기능명] -
6 /test-design [기능명] -
7 /prd-finalize [기능명] -
8 /implement [기능명] --backend-only, --frontend-only, --resume
💡 명령어 사용 팁
  • 기능명은 kebab-case 사용 (예: user-management, notification-system)
  • --fix 옵션은 Gate 실패 시 해당 Gate만 재실행
  • --no-multi-llm은 기존 기능 수정 시 단일 LLM 모드로 실행
  • --from [N]으로 특정 Gate부터 재시작 가능

🏷️ 8. Provisional 항목 관리

Provisional 항목이란?

표준 문서에 정의되지 않은 신규 항목으로, 워크플로우 진행을 위해 임시로 사용합니다.

유형 표준 문서 태그 발생 Gate
UI 컴포넌트 docs/ui-standards/UI_STANDARD.md [PROVISIONAL] Gate 2
디자인 트렌드 docs/ui-standards/UI_STANDARD.md [NEW TREND] Gate 2
DB 용어 db_dic/dictionary/standards.json [PROVISIONAL] Gate 3

승인 프로세스

Gate 2/3에서 발견
→
[PROVISIONAL] 태그
→
Gate 7에서 수집
→
사용자 승인
→
표준 문서 업데이트

승인 옵션

[A] 전체 승인

모든 Provisional 항목을 표준 문서에 추가합니다. UI_STANDARD.md 또는 standards.json이 자동 업데이트됩니다.

[P] 프로젝트 한정

이번 프로젝트에서만 사용합니다. PRD에만 기록되고 표준 문서는 변경되지 않습니다.

[S] 개별 선택

각 항목별로 A/P/R을 선택할 수 있습니다.

[R] 전체 거부

Provisional 항목을 기존 표준 항목으로 대체해야 합니다.

📚 9. 워크플로우 활용 가이드

언제 전체 워크플로우를 사용하나요?

✅ 사용해야 할 때

  • 완전히 새로운 도메인 기능 추가
  • 아키텍처에 큰 영향을 미치는 변경
  • 복잡한 비즈니스 로직 구현
  • 여러 화면/API가 연관된 기능

❌ 사용하지 않아도 될 때

  • 기존 기능의 버그 수정
  • UI/UX 개선
  • 성능 최적화
  • 기존 기능에 필드/버튼 추가

에러 처리 가이드

서브에이전트 실패 시 ▼
  1. 실패한 Gate 식별
  2. 오류 메시지 확인
  3. 해당 서브에이전트만 재실행: /design-all {기능명} --from {실패한 Gate 번호}
80% 완성도 미달 시 ▼
  1. Gate 7에서 미달 항목 리스트 확인
  2. 해당 Gate 서브에이전트 재실행 (예: /api-design --fix)
  3. /design-all {기능명} --from 7로 재검증
빌드/테스트 실패 시 ▼
  1. 전체 오류 로그 확인
  2. 의존성 문제 확인
  3. 순차적으로 수정
  4. /implement {기능명} --resume로 재개

코드 아키텍처 규칙

Backend 규칙
  • Service 클래스는 300줄 이하 유지
  • 새 도메인 기능은 별도 Service 클래스로 생성
  • 책임 분리: Service(비즈니스) / Mapper(데이터) / Client(외부 API)
Frontend 규칙
  • 컴포넌트 파일은 500줄 이하 유지
  • 공통 로직은 hooks/로 분리
  • 유틸리티 함수는 utils/로 분리
  • UI_STANDARD.md 준수 필수
